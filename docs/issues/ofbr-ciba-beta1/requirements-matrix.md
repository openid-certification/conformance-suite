## Variant Decision

Default to the existing `openbanking_brazil` profile. Create a separate beta1 variant only if this matrix reveals legacy Brazil CIBA behavior that must remain certifiable separately or if beta1 semantics would break existing published configurations.

## DCR/DCM delivery status

| OP / AS under test | Directory token and SSA retrieval, OFBR mTLS registration with `jwks_uri`, CIBA ping metadata, response validation, and registered-client use | Brazil `dynamic_client` completes the full Data Consent, ping, token, encrypted ID Token, and `/resources` flow | G1 decides whether static Brazil scheduling remains additive or is rejected plan-wide. G2 decides whether to publish the deliberate `backchannel_user_code_parameter=true` initial-DCR test. |
| RP / client under test | Dedicated Brazil mTLS DCR endpoint and happy-path module validating Directory SSA, OFBR metadata, CIBA ping metadata, signing and encryption keys | `fapi-ciba-id1-client-brazildcr-happypath-test` is published and paired with the OP dynamic-client happy module | The standard suite interaction cannot force an external RP to issue a DCM PUT. |
| DCM against an AS under test | Existing registration-management primitives make a full authenticated PUT technically feasible after successful DCR | Not published and not claimed by cleanup DELETE | G3 must explicitly activate the OP-side PUT test. There is no symmetric RP-side claim. |

The focused CI entry pairs `fapi-ciba-id1-client-brazildcr-happypath-test` with the Brazil `dynamic_client` variant of `fapi-ciba-id1-ensure-other-scope-order-succeeds`. Its discovered rerun number is intentionally not documented because runner numbering changes as pairings are edited.
